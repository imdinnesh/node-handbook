# Asynchronous Flow Control

> [!NOTE]
> The material in this post is heavily inspired by Mixu's Node.js Book.

At its core, JavaScript is designed to be non-blocking on the "main" thread—this is where views are rendered. In the browser, when the main thread becomes blocked, it results in the infamous "freezing" that users dread. No other events can be dispatched, resulting in data loss and a poor user experience.

This creates unique constraints that only a functional style of programming can solve. This is where **callbacks** come into the picture.

---

## The Problem: Callback Hell

Callbacks can become challenging to handle in complicated procedures. This often results in **"callback hell"**, where multiple nested functions make the code difficult to read, debug, and organize.

```javascript
async1(function (input, result1) {
  async2(function (result2) {
    async3(function (result3) {
      async4(function (result4) {
        async5(function (output) {
          // Do something with output
        });
      });
    });
  });
});
```

In real-world scenarios, additional lines of code to handle `result1`, `result2`, etc., make this issue much messier than the simple example above.

---

## Anatomy of Asynchronous Operations

Complex operations are often made up of three types of functions:

1.  **Initiator**: The first function in the sequence. It accepts the original input (global variables, direct invocation, or data from FS/Network).
2.  **Middleware**: Returns another function.
3.  **Terminator**: Invokes the callback.

### Example Flow
The following illustrates a flow where latency is 0 because values are available in memory:

```javascript
function final(someInput, callback) {
  callback(`${someInput} and terminated by executing callback `);
}

function middleware(someInput, callback) {
  return final(`${someInput} touched by middleware `, callback);
}

function initiate() {
  const someInput = 'hello this is a function ';
  middleware(someInput, function (result) {
    console.log(result);
  });
}

initiate();
```

## State Management

Functions may or may not be **state-dependent**. State dependency arises when a function's logic relies on an outside variable or function.

There are two primary strategies for managing state:
1.  **Direct Passing**: Passing variables directly as arguments.
2.  **External Sources**: Acquiring values from a cache, session, file, database, or network.

> [!WARNING]
> Managing state with **global variables** is often a sloppy anti-pattern. It makes it difficult to guarantee state consistency. Global variables should be avoided in complex programs whenever possible.

## Control Flow

### 1. In-Memory Iteration
If an object is available in memory, sync-style iteration works fine:

```javascript
function getSong() {
  let _song = '';
  for (let i = 100; i > 0; i -= 1) {
    _song += `${i} beers on the wall...\n`;
    if (i === 1) _song += "Hey let's get some more beer";
  }
  return _song;
}

function singSong(_song) {
  if (!_song) throw new Error("song is empty!");
  console.log(_song);
}

const song = getSong();
singSong(song); // This works!
```

### 2. Outside-Memory (Asynchronous)
If the data exists outside of memory, standard iteration fails:

```javascript
function getSong() {
  let _song = '';
  for (let i = 100; i > 0; i -= 1) {
    setTimeout(function () {
      _song += `${i} beers on the wall...\n`;
    }, 0);
  }
  return _song;
}

const song = getSong();
singSong(song); 
// Error: song is empty!
```

**Why did this happen?**
`setTimeout` instructs the CPU to store instructions elsewhere and schedules them for later execution. Thousands of CPU cycles pass before the timer triggers. However, the `return _song` statement executed thousands of cycles *prior* to the `song` being updated.

## The 3 Core Async Patterns

You can perform almost all operations using these three patterns:

### 1. In Series
Functions execute in a strict sequential order, similar to a `for` loop but for async tasks.

```javascript
const operations = [
  { func: function1, args: args1 },
  { func: function2, args: args2 },
  { func: function3, args: args3 },
];

function serialProcedure(operation) {
  if (!operation) process.exit(0); // Finished
  
  operation.func(operation.args, function (result) {
    // Continue AFTER callback
    serialProcedure(operations.shift());
  });
}

serialProcedure(operations.shift());
```

### 2. Limited In Series
Sequential execution but with a cap. Useful for processing large lists with a target success count.

```javascript
let successCount = 0;

function serial(recipient) {
  if (!recipient || successCount >= 1000000) {
    return final();
  }
  dispatch(recipient, function (err) {
    if (!err) successCount += 1;
    serial(bigList.pop());
  });
}
```

### 3. Full Parallel
Used when ordering is not an issue (e.g., sending 1,000,000 emails simultaneously).

```javascript
recipients.forEach(function (recipient) {
  dispatch(recipient, function (err) {
    if (!err) {
      success += 1;
    } else {
      failed.push(recipient.name);
    }
    count += 1;
    if (count === recipients.length) {
      final({ count, success, failed });
    }
  });
});
```

## Summary Best Practices

*   **Modularize**: Break procedures into small, focused functions.
*   **Use Callbacks**: Don't fear them—just manage them well.
*   **Think Middleware**: If you're unsure of how to structure a function, treat it as if it were middleware!
