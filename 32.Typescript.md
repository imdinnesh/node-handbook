# Introduction to TypeScript

TypeScript is an open-source language maintained and developed by Microsoft. It adds additional syntax to JavaScript to support a tighter integration with your editor, allowing you to catch errors early in your editor or in your CI/CD pipeline, resulting in more maintainable code.

## What is TypeScript?

TypeScript is basically JavaScript with types. It helps you:
- Catch errors during development.
- Write more predictable and maintainable code.
- Improve developer experience with better autocompletion.

## First TypeScript Code

Take a look at this code snippet:

```typescript
type User = {
  name: string;
  age: number;
};

function isAdult(user: User): boolean {
  return user.age >= 18;
}

const justine = {
  name: 'Justine',
  age: 23,
} satisfies User;

const isJustineAnAdult = isAdult(justine);
```

### Unpacking the Example
- **The `type` keyword**: Declares our custom object type representing users.
- **Function Type Annotations**: The `isAdult` function accepts one argument of type `User` and returns a `boolean`.
- **`satisfies` operator**: Ensures `justine` matches the `User` type.

> [!NOTE]
> - If we do not comply with declared types, TypeScript will inform us that something is wrong and prevent misuse.
> - Not everything must be typed explicitly—TypeScript **infers** types for us (e.g., `isJustineAnAdult` is inferred as a `boolean`).

## What does TypeScript consist of?

TypeScript consists of two main components:

### 1. TypeScript Code
Regular JavaScript with additional syntax for type annotations. When compiled, all TypeScript-specific parts are removed, resulting in clean JavaScript.

```typescript
function greet(name: string) {
  console.log(`Hello, ${name}!`);
}
```

### 2. Type Definitions
Type definitions describe the "shape" of existing JavaScript code. They are usually stored in `.d.ts` files and don't contain implementation—only descriptions of types.

For Node.js, you'll need type definitions for its APIs, available via `@types/node`.

```bash
npm add --save-dev @types/node
```

These definitions enable autocompletion for Node.js functions like `fs.readFile` or `http.createServer`.

```typescript
// @errors: 2345
import { resolve } from 'node:path';
resolve(123, 456); // Error: Argument of type 'number' is not assignable to parameter of type 'string'.
```

> [!TIP]
> Many popular libraries have type definitions under the `@types` namespace, maintained by the **DefinitelyTyped** community.

### Transform Capabilities
TypeScript can also transform modern syntax like **JSX** (used in React) into regular JavaScript, acting as a build tool similar to Babel.

## How to run TypeScript code

There are three primary ways to execute TypeScript:

### Running TypeScript Natively
Modern Node.js (v22.18.0+) can execute TypeScript directly if it contains only erasable syntax.

- **Node.js v22.18.0+**:
  ```bash
  node example.ts
  ```
- **Earlier versions (but still recent)**:
  ```bash
  node --experimental-strip-types example.ts
  ```

> [!IMPORTANT]
> - You can disable this via `--no-experimental-strip-types`.
> - For syntax like `enums` or `namespaces`, use `--experimental-transform-types` (introduced in v22.7.0). Enabling this automatically implies `--experimental-strip-types`.

### Constraints
The support for TypeScript in Node.js has some constraints to keep in mind. You can get more information on the [API docs](https://nodejs.org/api/typescript.html).

### Configuration
The Node.js TypeScript loader (Amaro) does **not** need or use `tsconfig.json` to run TypeScript code.

However, we recommend configuring your editor and `tsc` to reflect Node.js behavior by creating a `tsconfig.json` using the recommended compiler options and using TypeScript version 5.7 or higher.

### Running TypeScript using Transpilation
Transpilation is the process of converting TypeScript to JavaScript.

1. **Install TypeScript**:
   ```bash
   npm i -D typescript
   ```
2. **Compile**:
   ```bash
   npx tsc example.ts
   ```
   *This creates `example.js`.*
3. **Run**:
   ```bash
   node example.js
   ```

> [!NOTE]
> `npx` allows you to run packages without global installation. `tsc` is the TypeScript compiler.

#### Catching Type Errors
If you introduce an error, the compiler will stop you:

```typescript
// @errors: 2322 2554
type User = {
  name: string;
  age: number;
};

const justine: User = {
  name: 'Justine',
  age: 'Secret!', // Error: Type 'string' is not assignable to type 'number'.
};
```

### Running TypeScript with a Runner
If you need advanced features or are on an older Node.js version:

- **`ts-node`**: Performs type checking by default.
  ```bash
  npm i -D ts-node
  npx ts-node example.ts
  ```
- **`tsx`**: Fast, but does **not** type check.
  ```bash
  npm i -D tsx
  npx tsx example.ts
  ```
  *Register via node:* `node --import=tsx example.ts`

## Publishing a TypeScript Package

When publishing to npm, you distribute **JavaScript** and **Type Definitions**, not raw source code.

### Key Considerations:
- **`main` field**: Points to the compiled `.js` file (e.g., `"main": "dist/index.js"`).
- **`scripts.test`**: Can point to `.ts` files if using a runner.
- **Node.js behavior**: Node runs TypeScript via "type stripping" (removing TypeScript syntax). This is enabled by default in v22.18.0+.
- **`node_modules`**: Node does **not** strip types in `node_modules` for performance reasons. TypeScript maintainers discourage publishing raw TypeScript.
- **Erasable Syntax**: To ensure your code can run in Node without flags, set the `erasableSyntaxOnly` config option in TypeScript 5.8+.
- **`tsconfig.json`**: Use `dependabot` to keep dependencies current and `.nvmrc` to specify the Node.js version.

### Repository Structure Example:
```text
example-ts-pkg/
├ .github/
│ ├ workflows/
│ │ ├ ci.yml
│ │ └ publish.yml
│ └ dependabot.yml
├ src/
│ ├ foo.fixture.js
│ ├ main.ts
│ ├ main.test.ts
│ ├ some-util.ts
│ └ some-util.test.ts
├ LICENSE
├ package.json
├ README.md
└ tsconfig.json
```

### Published Package Structure:
```text
example-ts-pkg/
├ LICENSE
├ main.d.ts
├ main.d.ts.map
├ main.js
├ package.json
├ README.md
├ some-util.d.ts
├ some-util.d.ts.map
└ some-util.js
```

## What to do with your types

### Treat types like a test
Types warn you when an implementation won't work. They are complementary to unit tests.

```typescript
// @errors: 2322
const foo = 'a';
const bar: number = 1 + foo; // Warning: Operator '+' cannot be applied to types 'number' and 'string'.
```

#### Automating with GitHub Actions
You can set up CI to automatically check (and require) types pass inspection for a PR into the main branch.

```yaml
# yaml-language-server: $schema=https://json.schemastore.org/github-workflow.json
name: Tests
on:
  pull_request:
    branches: ['*']
jobs:
  check-types:
    # Separate these from tests because they are platform 
    # and node-version independent and need be run only once.
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version-file: '.nvmrc'
          cache: 'npm'
      - name: npm clean install
        run: npm ci
      # You may want to run a lint check here too
      - run: node --run types:check

  get-matrix:
    # Automatically pick active LTS versions
    runs-on: ubuntu-latest
    outputs:
      latest: ${{ steps.set-matrix.outputs.requireds }}
    steps:
      - uses: ljharb/actions/node/matrix@main
        id: set-matrix
        with:
          versionsAsRoot: true
          type: majors
          preset: '>= 22' # glob is not backported below 22.x

  test:
    needs: [get-matrix]
    runs-on: ${{ matrix.os }}
    strategy:
      fail-fast: false
      matrix:
        node-version: ${{ fromJson(needs.get-matrix.outputs.latest) }}
        os:
          - macos-latest
          - ubuntu-latest
          - windows-latest
    steps:
      - uses: actions/checkout@v4
      - name: Use node ${{ matrix.node-version }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'npm'
      - name: npm clean install
        run: npm ci
      - run: node --run test
```

### Generate type declarations
Type declarations (`.d.ts`) allow your code to be vanilla JavaScript while still providing type info to consumers.

> [!TIP]
> Generate these as part of your publication process using `prepack`. You don't need to check them into Git.

```yaml
name: Publish to npm
on:
  push:
    tags:
      - '**@*'
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version-file: '.nvmrc'
          registry-url: 'https://registry.npmjs.org'
      - run: npm ci
      # npm publish will run prepack (generating .d.ts) automatically
```

### Detailed Breakdown
- **Deterministic Output**: Generating type declarations is deterministic: you'll get the same output from the same input every time. There is no need to commit these to git.
- **Publication Process**: `npm publish` grabs everything available at the moment it's run. Generating declarations immediately before (via `prepack`) ensures they are included.
- **Node.js LTS Support**: You should publish packages compiled to support all Node.js LTS versions. The configurations in this article support Node 18.x and later.
- **`prepack` Behavior**: `npm publish` and `npm pack --dry-run` automatically run `prepack`. Note that `node --run` does **not** do this.

---

### Minimizing Package Size
By default, `npm publish` includes almost everything. To keep your package minimal, exclude tests and fixtures using `.npmignore`. 

> [!CAUTION]
> If using `.npmignore`, ensure the `!*.d.ts` exception is listed, or your generated type declarations won't be published! Alternatively, use the `files` field in `package.json`.

---

> [!IMPORTANT]
> Ensure your `.npmignore` doesn't accidentally block `.d.ts` files! Use a `!*.d.ts` exception if needed.
