# Working with Different Filesystems in Node.js

Node.js exposes many features of the filesystem, but not all filesystems are alike. The following are suggested best practices to keep your code simple and safe when working with different filesystems.

---

## 1. Understanding Filesystem Behavior

Before working with a filesystem, you must understand how it behaves. Different systems have varying support for:
- **Case sensitivity** vs. **Case insensitivity**
- **Case preservation**
- **Unicode form preservation**
- **Timestamp resolution** (nanoseconds vs. seconds)
- Extended attributes, inodes, Unix permissions, and alternate data streams.

> [!WARNING]
> **Avoid inferring behavior from `process.platform`.**
> - Do not assume **Darwin** (macOS) always uses a case-insensitive filesystem (HFS+); a user might use HFSX (case-sensitive).
> - Do not assume **Linux** always supports Unix permissions and inodes; it could be an external drive (FAT/NTFS) that does not.

### Probing Over Hardcoding
Instead of maintaining an incomplete list of every filesystem, **probe the filesystem** to see how it actually behaves. Testing for easily detectable features often reveals the behavior of more complex ones.

---

## 2. Avoid the "Lowest Common Denominator" Approach

You might be tempted to normalize everything (filenames to uppercase, timestamps to 1-second resolution, etc.) to make the program "portable."

> [!IMPORTANT]
> **Do not do this.** 
> Using a lowest common denominator approach makes your program leaky and prone to data corruption:
> - You will be unable to work with advanced filesystems as users expect.
> - You will run into filename or timestamp collisions.
> - You risk corrupting user data through complex dependent events.

---

## 3. The "Superset" Approach (Recommended)

Make the best use of each platform by adopting a **superset approach**. Your program should act like a more advanced filesystem by supporting the union of all possible features:
- Case-sensitivity and Case-preservation
- Unicode form sensitivity and Preservation
- Unix permissions
- High-resolution (nanosecond) timestamps
- Extended attributes

> [!TIP]
> Once you have **case-preservation** in your program, you can always implement case-insensitivity if needed. However, if you discard case-preservation, you cannot safely interact with a case-preserving filesystem.

---

## 4. Preservation Principles

### Case Preservation
You may create a directory called `test/abc` and find that `fs.readdir('test')` returns `['ABC']`. 
> [!NOTE]
> This is **not a bug in Node.js**. Node returns the filename exactly as the filesystem stores it. Some filesystems automatically convert filenames to uppercase or lowercase.

### Unicode Form Preservation
Unicode can encode the same characters using different byte sequences. Several strings may *look* the same but have different representations.

> [!EXAMPLE]
> Creating `test/café` (NFC form, length 5, `<63 61 66 c3 a9>`) might result in `fs.readdir` returning `['café']` (NFD form, length 6, `<63 61 66 65 cc 81>`).
> This is common on **HFS+**, which normalizes filenames to a form similar to NFD.

### Unicode Form Insensitivity
Unicode form insensitivity and preservation are often confused. It is better to implement insensitivity by using Unicode normalization for **comparison only**, without sacrificing the original byte sequences during storage.

---

## 5. Comparing Different Unicode Forms

Node.js provides `string.normalize('NFC' / 'NFD')` to normalize UTF-8 strings.

> [!CAUTION]
> **Never store the output of `normalize()`.** Use it only within comparison functions to check if two strings look the same to a user.

```javascript
// Recommended comparison function
const areStringsEquivalent = (s1, s2) => {
  return s1.normalize('NFC') === s2.normalize('NFC');
};
```

#### Optimization Tip
Normalization is fast, but for high-frequency operations, use a **local cache** to avoid re-normalizing the same strings. Do not persist this cache.

> [!NOTE]
> `normalize()` requires **ICU support** in Node.js. Standard official downloads include this by default.

---

## 6. Timestamp Resolution

Setting a file's `mtime` (modified time) to `1444291759414` (milliseconds) might result in `fs.stat` returning `1444291759000` (seconds) or `1444291758000` (2-second resolution).

This variation is dictated by the filesystem:
- **FAT filesystems**: May have coarse resolution (e.g., 24 hours for `atime`).
- **Modern filesystems**: Support nanosecond resolution.

---

## 7. Protecting User Data

### Do Not Corrupt Filenames and Timestamps
Filenames and timestamps are **user data**. You should never automatically rewrite them through normalization, just as you wouldn't automatically uppercase the contents of a text file.

> [!IMPORTANT]
> **Normalization is effectively a lossy hash code.** Use it for equivalence testing, but never as a substitute for actual data. Your program should pass on filename and timestamp data as-is.

### Appropriate Comparison
- Do not use case-insensitive comparison on a case-sensitive filesystem.
- Do not use Unicode-form-insensitive comparison on a sensitive filesystem (like NTFS or most Linux systems).
- Do not downsample timestamps to 2-second resolution if the filesystem supports nanoseconds.

---

## 8. Identifying Comparison Differences

Be aware that filesystem comparison logic can be subtle:
- **toUpperCase() vs. toLowerCase()**: In many systems, `toUpperCase()` handles foreign characters better than `toLowerCase()`.
- **HFS+ Quirks**: Apple's HFS+ use an older version of the NFD form which may differ slightly from the latest Unicode standard.

**Best Practice:** Whenever possible, **probe the filesystem** because every system has its own specific case comparison tables baked in.
