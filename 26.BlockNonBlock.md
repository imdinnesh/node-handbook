# Overview of Blocking vs Non-Blocking

This overview covers the difference between **blocking** and **non-blocking** calls in Node.js. It refers to the event loop and `libuv`, but no prior knowledge of those topics is required.

> [!NOTE]
> **"I/O"** refers primarily to interaction with the system's disk and network, supported by `libuv`.

---

## Blocking

**Blocking** is when the execution of additional JavaScript in the Node.js process must wait until a non-JavaScript operation completes. This happens because the event loop is unable to continue running JavaScript while a blocking operation is occurring.

*   In Node.js, JavaScript that exhibits poor performance due to being **CPU intensive** (rather than waiting on I/O) isn't typically referred to as blocking.
*   Synchronous methods in the Node.js standard library that use `libuv` are the most commonly used blocking operations.
*   Native modules may also have blocking methods.

All I/O methods in the Node.js standard library provide asynchronous versions, which are **non-blocking** and accept callback functions. Some methods also have blocking counterparts, which have names that end with `Sync`.

---

## Comparing Code

Blocking methods execute **synchronously**, and non-blocking methods execute **asynchronously**.

### Synchronous Example (Blocking)
Using the File System (`fs`) module, here is a synchronous file read:

```javascript
const fs = require('node:fs');
const data = fs.readFileSync('/file.md'); // blocks here until file is read
console.log(data);
moreWork(); // will run after console.log
```

### Asynchronous Example (Non-Blocking)
Here is an equivalent asynchronous example:

```javascript
const fs = require('node:fs');
fs.readFile('/file.md', (err, data) => {
  if (err) {
    throw err;
  }
  console.log(data);
});
moreWork(); // will run BEFORE console.log
```

> [!IMPORTANT]
> **Key Differences:**
> 1. In the first example, `moreWork()` waits for the file read. In the second, it runs immediately without waiting.
> 2. In the synchronous version, errors must be caught with `try/catch` or the process will crash. In the asynchronous version, errors are passed to the callback.

---

## Concurrency and Throughput

JavaScript execution in Node.js is **single-threaded**, so concurrency refers to the event loop's capacity to execute JavaScript callback functions after completing other work.

A key design choice in Node.js is to run non-JavaScript operations (like I/O) as non-blocking tasks. This allows for significantly higher throughput.

**Example Scenario:**
- Each request to a web server takes 50ms.
- 45ms of that is database I/O.
- By using **non-blocking** asynchronous operations, we free up that 45ms per request to handle other incoming requests.

> [!TIP]
> This is a fundamental difference between Node.js and other models where additional threads might be created to handle concurrent work.

---

## Dangers of Mixing Blocking and Non-Blocking Code

Mixing these two patterns can lead to bugs and race conditions. Consider this dangerous pattern:

```javascript
const fs = require('node:fs');

fs.readFile('/file.md', (err, data) => {
  if (err) throw err;
  console.log(data);
});

fs.unlinkSync('/file.md'); // DANGER: This might run before readFile is finished!
```

In the example above, `fs.unlinkSync()` (which deletes the file) is likely to run **before** `fs.readFile()` finishes its I/O, causing an error because the file is deleted before it can be read.

### The Correct Way (Completely Non-Blocking)
To guarantee the correct order, nest the dependent operation within the callback:

```javascript
const fs = require('node:fs');

fs.readFile('/file.md', (readFileErr, data) => {
  if (readFileErr) throw readFileErr;
  
  console.log(data);
  
  // Now it is safe to delete
  fs.unlink('/file.md', (unlinkErr) => {
    if (unlinkErr) throw unlinkErr;
  });
});
```
