# Don't Block the Event Loop (or the Worker Pool)

### Should you read this guide?
If you're writing anything more complicated than a brief command-line script, reading this should help you write higher-performance, more-secure applications.

> [!NOTE]
> This document is written with Node.js servers in mind, but the concepts apply to complex Node.js applications as well. Where OS-specific details vary, this document is Linux-centric.

---

## Summary

Node.js runs JavaScript code in the **Event Loop** (initialization and callbacks), and offers a **Worker Pool** to handle expensive tasks like file I/O. Node.js scales well, sometimes better than more heavyweight approaches like Apache. 

The secret to the scalability of Node.js is that it uses a small number of threads to handle many clients. If Node.js can make do with fewer threads, then it can spend more of your system's time and memory working on clients rather than on paying space and time overheads for threads (memory, context-switching). 

> [!IMPORTANT]
> **Rule of Thumb:** Node.js is fast when the work associated with each client at any given time is **"small"**. 
> 
> This applies to:
> - Callbacks on the Event Loop
> - Tasks on the Worker Pool

---

## Why should I avoid blocking the Event Loop and the Worker Pool?

Node.js uses a small number of threads to handle many clients. In Node.js there are two types of threads: 
1. **One Event Loop** (aka the main loop, main thread, event thread, etc.)
2. **A pool of k Workers** in a Worker Pool (aka the threadpool)

If a thread is taking a long time to execute a callback (Event Loop) or a task (Worker), we call it **"blocked"**. While a thread is blocked working on behalf of one client, it cannot handle requests from any other clients.

### Motivations for avoiding blocks:
- **Performance:** If you regularly perform heavyweight activity on either type of thread, the throughput (requests/second) of your server will suffer.
- **Security:** If it is possible that for certain input one of your threads might block, a malicious client could submit this "evil input", make your threads block, and keep them from working on other clients. This would be a **Denial of Service (DoS)** attack.

## A Quick Review of Node.js Architecture

Node.js uses the **Event-Driven Architecture**: it has an Event Loop for orchestration and a Worker Pool for expensive tasks.

### What code runs on the Event Loop?
When they begin, Node.js applications first complete an **initialization phase**, `require`'ing modules and registering callbacks for events. Node.js applications then enter the **Event Loop**, responding to incoming client requests by executing the appropriate callback. 

- This callback executes **synchronously**, and may register asynchronous requests to continue processing after it completes. 
- The callbacks for these asynchronous requests will also be executed on the Event Loop.
- The Event Loop will also fulfill the non-blocking asynchronous requests made by its callbacks, e.g., network I/O.

> [!TIP]
> In summary, the Event Loop executes the JavaScript callbacks registered for events and is also responsible for fulfilling non-blocking asynchronous requests like network I/O.

### What code runs on the Worker Pool?
The Worker Pool of Node.js is implemented in **libuv**, which exposes a general task submission API. Node.js uses the Worker Pool to handle "expensive" tasks. This includes I/O for which an operating system does not provide a non-blocking version, as well as particularly CPU-intensive tasks.

#### Core APIs using the Worker Pool:

| Category | API / Modules |
| :--- | :--- |
| **I/O-intensive** | **DNS:** `dns.lookup()`, `dns.lookupService()` |
| | **File System:** All file system APIs except `fs.FSWatcher()` and explicitly synchronous ones. |
| **CPU-intensive** | **Crypto:** `crypto.pbkdf2()`, `crypto.scrypt()`, `crypto.randomBytes()`, `crypto.randomFill()`, `crypto.generateKeyPair()` |
| | **Zlib:** All zlib APIs except those that are explicitly synchronous. |

> [!NOTE]
> Applications and modules that use a C++ add-on can submit other tasks to the Worker Pool.

## Task Management & Scheduling

### How does Node.js decide what code to run next?
Abstractly, the Event Loop and the Worker Pool maintain queues for pending events and pending tasks, respectively.

- **Event Loop:** Does not actually maintain a queue. Instead, it has a collection of **file descriptors** that it asks the OS to monitor (using epoll, kqueue, etc.). When the OS says a descriptor is ready, the Event Loop translates it to the appropriate event and invokes the associated callback(s).
- **Worker Pool:** Uses a **real queue** whose entries are tasks to be processed. A Worker pops a task from this queue and works on it. When finished, the Worker raises an "At least one task is finished" event for the Event Loop.

### What does this mean for application design?
In a one-thread-per-client system like Apache, each pending client is assigned its own thread. If a thread blocks, the OS interrupts it and gives another client a turn. 

**Because Node.js handles many clients with few threads:**
- If a thread blocks, pending client requests may not get a turn until the thread finishes.
- **Fair treatment of clients is the responsibility of your application.**
- You shouldn't do too much work for any client in any single callback or task.

## Don't Block the Event Loop

The Event Loop orchestrates everything. If it spends too long at any point, all current and new clients will not get a turn.

> [!CAUTION]
> You should make sure you never block the Event Loop. Each of your JavaScript callbacks should complete quickly. This also applies to `await`'s, `Promise.then`'s, and so on.

### Computational Complexity
A good way to ensure fairness is to reason about the "computational complexity" of your callbacks.

#### Example 1: Constant-time callback (O(1))
```javascript
app.get('/constant-time', (req, res) => {
  res.sendStatus(200);
});
```

#### Example 2: O(n) callback
Runs quickly for small `n` and more slowly for large `n`.
```javascript
app.get('/countToN', (req, res) => {
  const n = req.query.n;
  // n iterations before giving someone else a turn
  for (let i = 0; i < n; i++) {
    console.log(`Iter ${i}`);
  }
  res.sendStatus(200);
});
```

#### Example 3: O(n²) callback
Runs much slower than O(n) as `n` grows.
```javascript
app.get('/countToN2', (req, res) => {
  const n = req.query.n;
  // n^2 iterations before giving someone else a turn
  for (let i = 0; i < n; i++) {
    for (let j = 0; j < n; j++) {
      console.log(`Iter ${i}.${j}`);
    }
  }
  res.sendStatus(200);
});
```

### How careful should you be?
Node.js uses the Google V8 engine, which is very fast, but for complex tasks, you should bound the input and reject inputs that are too long. This ensures the callback cannot take more than the worst-case time on the longest acceptable input.

## Blocking the Event Loop: REDOS

One common way to block the Event Loop disastrously is by using a "vulnerable" regular expression.

### Avoiding Vulnerable Regular Expressions
A regular expression match can sometimes require an **exponential** number of trips through the input string — **O(2ⁿ)** time. This blocks the Event Loop.

A vulnerable regular expression is one where the engine might take exponential time, exposing you to **REDOS** on "evil input".

> [!WARNING]
> **Rules of thumb to avoid REDOS:**
> 1. Avoid nested quantifiers like `(a+)*`.
> 2. Avoid OR's with overlapping clauses, like `(a|a)*`.
> 3. Avoid using backreferences, like `(a.*) \1`.
> 4. For simple string matching, use `indexOf()` or equivalent.

### A REDOS Example
```javascript
app.get('/redos-me', (req, res) => {
  const filePath = req.query.filePath;
  // REDOS Vulnerability: Doubly-nested quantifier
  if (filePath.match(/(\/.+)+$/)) {
    console.log('valid path');
  } else {
    console.log('invalid path');
  }
  res.sendStatus(200);
});
```
If a client queries with `filePath` as `///.../\n` (100 `/`'s followed by a newline), the Event Loop will take effectively forever to mismatch, blocking all other clients.

### Anti-REDOS Resources
- [safe-regex](https://www.npmjs.com/package/safe-regex)
- [rxxr2](http://www.cs.bham.ac.uk/~hxt/research/rxxr2/)
- **Alternative Engine:** Use [node-re2](https://www.npmjs.com/package/re2), which uses Google's RE2 engine (guaranteed linear time, but check for compatibility).

## Blocking the Event Loop: Core Modules & JSON

### Node.js Core Modules
Several core modules have synchronous expensive APIs intended for scripting convenience, **not** for use in a server context:

- **Encryption:** `crypto.randomBytes` (Sync version), `crypto.randomFillSync`, `crypto.pbkdf2Sync`.
- **Compression:** `zlib.inflateSync`, `zlib.deflateSync`.
- **File System:** Avoid synchronous file system APIs (e.g., `fs.readFileSync`).
- **Child Process:** `spawnSync`, `execSync`, `execFileSync`.

### JSON DOS
`JSON.parse` and `JSON.stringify` are O(n) but can take a long time for large `n`.

> [!CAUTION]
> If your server manipulates large JSON objects (especially from clients), be cautious. A 50MB string can take ~0.7s to stringify and ~1.3s to parse, blocking the loop the entire time.

#### Example: JSON blocking
```javascript
let obj = { a: 1 };
const iterations = 20;

// Expand the object exponentially
for (let i = 0; i < iterations; i++) {
  obj = { obj1: obj, obj2: obj };
}

console.time('JSON.stringify');
const jsonString = JSON.stringify(obj);
console.timeEnd('JSON.stringify');

console.time('JSON.parse');
const parsed = JSON.parse(jsonString);
console.timeEnd('JSON.parse');
```

**Asynchronous JSON APIs:**
- `JSONStream`
- `Big-Friendly JSON` (BFJ)

## Complex Calculations Without Blocking

### 1. Partitioning
Partition your calculations so each runs on the Event Loop but regularly "yields" to other pending events.

#### Example: Partitioned Average (O(n) total, each step O(1))
```javascript
function asyncAvg(n, avgCB) {
  let sum = 0;
  function help(i, cb) {
    sum += i;
    if (i == n) {
      cb(sum);
      return;
    }
    // "Asynchronous recursion" via setImmediate
    setImmediate(help.bind(null, i + 1, cb));
  }
  
  help(1, function (sum) {
    const avg = sum / n;
    avgCB(avg);
  });
}

asyncAvg(1000000, (avg) => console.log('avg:', avg));
```

### 2. Offloading
For complicated tasks, move the work onto a **Worker Pool** to benefit from multiple cores.

- **Option A:** Built-in Worker Pool (C++ addon via N-API).
- **Option B:** Custom Worker Pool (Child Process or Cluster).

> [!WARNING]
> Do not create a Child Process for every client. Your server might become a "fork bomb". Use a Pool.

#### Downside of Offloading
Incurs **communication overhead**. You must serialize and deserialize objects to pass them between the Event Loop and Workers.

## Don't Block the Worker Pool

Each Worker completes its current Task before proceeding to the next. Your goal is to **minimize the variation in Task times** using **Task partitioning**.

### Why minimize variation?
If one Task is much more expensive than others, it makes that Worker unavailable for a long time, effectively decreasing the size of the Pool and lowering overall throughput.

#### Examples of Variation:
1. **Long-running FS reads:** Using `fs.readFile()` on very large files (it wasn't partitioned before v10). 
   - *Fix:* Use `fs.read()` (manual partitioning) or `ReadStream`.
2. **Long-running Crypto:** `crypto.randomBytes()` is not partitioned. Requesting huge amounts of bytes at once will block a Worker.

### Task Partitioning in Workers
Partition each Task into comparable-cost sub-Tasks.
- Shorter tasks = few sub-tasks.
- Longer tasks = many sub-tasks.
- This allows a Worker to interleave sub-tasks from different clients.

## The Risks of NPM Modules

Third-party modules are great but often come with only "best-effort" guarantees.

> [!IMPORTANT]
> **Check for two things:**
> 1. Does it honor its APIs?
> 2. Might its APIs block the Event Loop or a Worker?

Even if an API is asynchronous, it might spend too much time in a single "partition" on the Event Loop or Worker, making it unsafe for large inputs.

---

## Conclusion

Node.js has two types of threads: **one Event Loop** and **k Workers**. 
- **Event Loop:** JavaScript callbacks and non-blocking I/O.
- **Workers:** Blocking I/O and CPU-intensive work.

If any callback or task takes a long time, the thread becomes blocked. This leads to degraded throughput or complete Denial of Service. 

**To write a high-performance, DoS-proof server, ensure that neither your Event Loop nor your Workers will block, even on malicious input.**
