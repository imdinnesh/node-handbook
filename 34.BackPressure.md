# Backpressuring in Streams

Backpressure is a fundamental concept in data handling that describes a build-up of data behind a buffer during transfer. When the receiver is slower than the sender—due to complex operations or resource constraints—incoming data accumulates like a clog.

To ensure a smooth flow, a delegation system must be in place. While Unix pipes and TCP sockets use "flow control," Node.js uses **Backpressure** within its Streams API.

## The Problem with Data Handling

In Node.js, data is transferred via **Streams**. Almost every internal module utilizes them, and they are highly recommended for application code.

### Basic Stream Usage
```javascript
// CJS / ESM
const readline = require('node:readline');

// process.stdin and process.stdout are both instances of Streams.
const rl = readline.createInterface({
  input: process.stdin,
  output: process.stdout,
});

rl.question('Why should you use streams? ', answer => {
  console.log(`Maybe it's ${answer}, maybe it's because they are awesome! :)`);
  rl.close();
});
```

A common optimization example is comparing standard tools with Node.js Streams. For instance, zipping a large file (~9 GB):

```javascript
// CJS / ESM
const fs = require('node:fs');
const gzip = require('node:zlib').createGzip();
const inp = fs.createReadStream('The.Matrix.1080p.mkv');
const out = fs.createWriteStream('The.Matrix.1080p.mkv.gz');

inp.pipe(gzip).pipe(out);
```

> [!IMPORTANT]
> In the example above, `.pipe()` lacks proper error handlers. If a chunk fails, the streams won't be destroyed automatically.

### Modern Alternative: `pipeline`
For Node.js 10.x+, the `pipeline` method is preferred over legacy `.pipe()` as it handles error forwarding and cleanup automatically.

**Using Callbacks:**
```javascript
const { pipeline } = require('node:stream');
const fs = require('node:fs');
const zlib = require('node:zlib');

pipeline(
  fs.createReadStream('The.Matrix.1080p.mkv'),
  zlib.createGzip(),
  fs.createWriteStream('The.Matrix.1080p.mkv.gz'),
  err => {
    if (err) {
      console.error('Pipeline failed', err);
    } else {
      console.log('Pipeline succeeded');
    }
  }
);
```

**Using Async/Await:**
```javascript
const { pipeline } = require('node:stream/promises');
const fs = require('node:fs');
const zlib = require('node:zlib');

async function run() {
  try {
    await pipeline(
      fs.createReadStream('The.Matrix.1080p.mkv'),
      zlib.createGzip(),
      fs.createWriteStream('The.Matrix.1080p.mkv.gz')
    );
    console.log('Pipeline succeeded');
  } catch (err) {
    console.error('Pipeline failed', err);
  }
}
```

## Too Much Data, Too Quickly

When a **Readable** stream produces data faster than a **Writable** consumer can handle, a queue forms. This queue consumes memory until the process completes.

> [!NOTE]
> Writing to disk is significantly slower than reading. Without backpressure, your system's memory remains monopolized, leading to:
> - System-wide slowdowns.
> - Overworked Garbage Collection (GC).
> - Potential memory exhaustion (OOM).

**The difference is an order of magnitude.** Backpressure allows Node.js to use a fixed, small amount of memory regardless of file size.

## How Does Backpressure Resolve These Issues?

Backpressure acts as a control loop between the **Source** (Readable) and the **Consumer** (Writable).

1.  **Trigger**: When a Writable stream's internal buffer exceeds the `highWaterMark` or the disk/network is busy, `.write(chunk)` returns `false`.
2.  **Pause**: The source is signaled to stop sending data.
3.  **Resume**: Once the Writable's buffer is cleared, it emits a `'drain'` event, signaling the Readable to resume.

> [!TIP]
> This mechanism ensures no memory leakage, no infinite buffering, and a predictable load for the Garbage Collector.

   |       |       |      |

## Backpressure Guidelines

Since Node.js v0.10, the `Stream` class allows modifying behavior via `._read()` and `._write()`.

### Rules to Abide By When Implementing Custom Streams
The **Golden Rule**: Always respect backpressure.

1.  **Never `.push()` if not requested**: Only push data when the consumer asks for it.
2.  **Respect `.write()` returns**: If it returns `false`, wait for the `'drain'` event.
3.  **Test for version variance**: Stream behavior can change across Node.js versions.

> [!TIP]
> For browser compatibility, consider use `readable-stream`. It provides automated graceful degradation and supports older environments.

---

### Rules specific to Readable Streams

A Readable stream is just as responsible for backpressure as a Writable one. It must respect the return value of `.push()`.

*   If `.push()` returns `false`, the stream must stop reading from the source.
*   If `.push()` returns `true`, it may continue.

#### ❌ Bad Practice (Ignoring Returns)
```javascript
class MyReadable extends Readable {
  _read(size) {
    let chunk;
    while (null !== (chunk = getNextChunk())) {
      this.push(chunk); // Problematic: ignoring backpressure signals!
    }
  }
}
```

#### ✅ Good Practice (Respecting Returns)
```javascript
class MyReadable extends Readable {
  _read(size) {
    let chunk;
    let canPushMore = true;
    while (canPushMore && null !== (chunk = getNextChunk())) {
      canPushMore = this.push(chunk);
    }
  }
}
```

### Rules specific to Writable Streams

When implementing or using Writable streams directly:

1.  **Wait for Drain**: If `.write()` returns `false`, do not call it again until `'drain'` is emitted.
2.  **HighWaterMark**: Be aware that large chunks exceeding `highWaterMark` will trigger backpressure.

#### Handling Callbacks Properly
```javascript
// Ensure only one callback is fired
class MyWritable extends Writable {
  _write(chunk, encoding, callback) {
    if (chunk.toString().indexOf('a') >= 0) {
      return callback(); // Use 'return' to prevent double-callback
    }
    if (chunk.toString().indexOf('b') >= 0) {
      return callback();
    }
    callback();
  }
}
```

#### Using `.cork()` and `.uncork()` Correctly
To optimize multiple small writes, use `process.nextTick()` to uncork:

```javascript
ws.cork();
ws.write('hello ');
ws.write('world ');
process.nextTick(doUncork, ws);

ws.cork();
ws.write('from ');
ws.write('Matteo');
process.nextTick(doUncork, ws);

function doUncork(stream) {
  stream.uncork();
}
```
