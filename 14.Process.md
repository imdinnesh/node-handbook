# Managing Environment Variables with `process.env`

The `process` core module in Node.js provides the `env` property, which contains all environment variables set at the moment the process was started.

---

## 1. Basic CLI Usage

You can set environment variables directly in the terminal when launching your application:

```bash
USER_ID=239482 USER_KEY=foobar node app.js
```

These variables are hosted on the global `process` object and can be accessed without any imports:

```javascript
console.log(process.env.USER_ID);  // "239482"
console.log(process.env.USER_KEY); // "foobar"
```

---

## 2. Native `.env` File Support (Node.js 20+)

Prior to Node.js 20, developers typically used third-party packages like `dotenv`. Node.js now includes native support for loading `.env` files via CLI flags.

### CLI Flags
To load a specific environment file, use the `--env-file` flag:

```bash
node --env-file=.env app.js
```

### Advanced Flag Options
*   **Multiple Files**: You can pass multiple flags. Subsequent files will override variables defined in previous ones.
    ```bash
    node --env-file=.env --env-file=.development.env app.js
    ```
*   **Optional Loading**: Use `--env-file-if-exists` to prevent the process from crashing if the specified file is missing.
    ```bash
    node --env-file-if-exists=.env app.js
    ```

> [!IMPORTANT]
> **Precedence Rule**: If the same variable is defined both in the system environment and in a `.env` file, the **system environment value takes precedence**.

---

## 3. Programmatic Loading

If you prefer to load environment files from within your code rather than via CLI flags, you can use the built-in `loadEnvFile` method.

### Example Usage
```javascript
const { loadEnvFile } = require('node:process');

// Loads from the default .env file or a custom path
loadEnvFile('./config/.env');

console.log(process.env.PORT);
```

### Key Differences from CLI Flags:
*   **Post-Initialization**: Since this is called within the code, variables that affect process startup (like `NODE_OPTIONS`) will **not** take effect, though they will still be accessible via `process.env`.
*   **Dynamic**: Allows for more complex logic when determining which environment file to load at runtime.
