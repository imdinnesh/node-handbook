# The Node.js Event Loop

The event loop is what allows Node.js to perform non-blocking I/O operations — despite being single-threaded — by offloading operations to the system kernel whenever possible.

Most modern kernels are multi-threaded; they can handle multiple operations executing in the background. When one of these operations completes, the kernel notifies Node.js so that the appropriate callback can be added to the **poll queue** to eventually be executed.

---

## How the Event Loop Works

When Node.js starts, it:
1. Initializes the event loop.
2. Processes the input script (which may make async API calls, schedule timers, or call `process.nextTick()`).
3. Begins processing the event loop.

### Lifecycle Diagram

The following diagram shows a simplified overview of the event loop's order of operations:

```text
   ┌───────────────────────────┐
   │           timers          │
   └─────────────┬─────────────┘
                 │
                 v
   ┌───────────────────────────┐
┌─>│     pending callbacks     │
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │       idle, prepare       │
│  └─────────────┬─────────────┘      ┌───────────────┐
│  ┌─────────────┴─────────────┐      │   incoming:   │
│  │           poll            │<─────┤  connections, │
│  └─────────────┬─────────────┘      │   data, etc.  │
│  ┌─────────────┴─────────────┐      └───────────────┘
│  │           check           │
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │      close callbacks      │
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
└──┤           timers          │
   └───────────────────────────┘
```

> [!NOTE]
> Each box is referred to as a **"phase"** of the event loop. Each phase has a FIFO (First-In-First-Out) queue of callbacks. When the loop enters a phase, it executes callbacks until the queue is exhausted or a system limit is reached.

---

## Phases of the Event Loop

| Phase | Description |
| :--- | :--- |
| **timers** | Executes callbacks scheduled by `setTimeout()` and `setInterval()`. |
| **pending callbacks** | Executes I/O callbacks deferred to the next loop iteration (e.g., TCP errors). |
| **idle, prepare** | Used only internally. |
| **poll** | Retrieves new I/O events; executes I/O related callbacks. Node will block here if appropriate. |
| **check** | `setImmediate()` callbacks are invoked here. |
| **close callbacks** | Executes close callbacks (e.g., `socket.on('close', ...)`). |

> [!IMPORTANT]
> Since Node.js 20 (libuv 1.45.0), the event loop runs timers only after the poll phase, instead of both before and after as in earlier versions.

---

## Phases in Detail

### Poll Phase
The poll phase has two main functions:
1. Calculating how long it should block and poll for I/O.
2. Processing events in the poll queue.

If the poll queue is **empty**:
- If scripts have been scheduled by `setImmediate()`, the loop ends the poll phase and continues to the **check** phase.
- If no `setImmediate()` scripts exist, the loop waits for callbacks to be added to the queue.

Once the poll queue is exhausted, the loop checks for timers that have reached their threshold and wraps back to the **timers** phase.

### Check Phase
This phase executes callbacks immediately after the poll phase completes. If the poll phase becomes idle and scripts have been queued with `setImmediate()`, the loop continues here rather than waiting.

### Timers Phase
A timer specifies the *threshold* after which a callback *may* be executed, not the exact time. OS scheduling or other callbacks can delay them.

**Example of Timer Delay:**
If you schedule a 100ms timeout but a file read operation takes 95ms and its callback takes another 10ms, the timer will fire at **105ms**, not 100ms.

```javascript
const fs = require('node:fs');

function someAsyncOperation(callback) {
  // Assume this takes 95ms to complete
  fs.readFile('/path/to/file', callback);
}

const timeoutScheduled = Date.now();

setTimeout(() => {
  const delay = Date.now() - timeoutScheduled;
  console.log(`${delay}ms have passed since I was scheduled`);
}, 100);

someAsyncOperation(() => {
  const startCallback = Date.now();
  // Simulate heavy work (10ms)
  while (Date.now() - startCallback < 10) { }
});
```

## `setImmediate()` vs `setTimeout()`

- **`setImmediate()`**: Designed to execute once the current **poll** phase completes.
- **`setTimeout()`**: Schedules a script to run after a minimum threshold (ms).

### Execution Order
The order depends on the context:
1. **Outside an I/O cycle**: The order is non-deterministic (depends on process performance).
2. **Inside an I/O cycle**: `setImmediate()` is **always** executed before `setTimeout(..., 0)`.

```javascript
// Inside an I/O cycle
const fs = require('node:fs');

fs.readFile(__filename, () => {
  setTimeout(() => console.log('timeout'), 0);
  setImmediate(() => console.log('immediate'));
});
// Output: 
// immediate
// timeout
```

## `process.nextTick()`

`process.nextTick()` is **not** technically part of the event loop. Instead, the `nextTickQueue` is processed after the current operation is completed, regardless of the current phase.

> [!CAUTION]
> Using recursive `process.nextTick()` calls can **starve your I/O** because it prevents the event loop from ever reaching the poll phase.

### Why use `process.nextTick()`?

1. **API Consistency**: Ensure an API is always asynchronous, even when returning an error immediately.
2. **Call Stack Unwinding**: Allow the script to run to completion (initializing variables, handlers) before a callback is executed.

**Example: Initialization Race Condition**
Without `nextTick`, the callback might try to access `bar` before it is assigned.

```javascript
let bar = null;

function someAsyncApiCall(callback) {
  process.nextTick(callback);
}

someAsyncApiCall(() => {
  console.log('bar', bar); // 1
});

bar = 1;
```

**Example: EventEmitter Constructors**
To emit an event from a constructor, you must wait until the user has had a chance to attach a listener.

```javascript
const EventEmitter = require('node:events');

class MyEmitter extends EventEmitter {
  constructor() {
    super();
    process.nextTick(() => {
      this.emit('event');
    });
  }
}

const myEmitter = new MyEmitter();
myEmitter.on('event', () => {
  console.log('an event occurred!');
});
```

## Comparison: `process.nextTick()` vs `setImmediate()`

- **`process.nextTick()`**: Fires **immediately** on the same phase (before the event loop continues).
- **`setImmediate()`**: Fires on the **following iteration** or 'tick' of the event loop.

> [!TIP]
> Use `setImmediate()` in most cases as it is easier to reason about and avoids I/O starvation.
