# Understanding `setImmediate()`

To execute code asynchronously and as soon as possible in Node.js, you can use `setImmediate()`.

```javascript
setImmediate(() => {
  // Executes in the next iteration of the event loop
});
```

---

## Comparison with Other Asynchronous Methods

How does `setImmediate()` differ from `setTimeout(..., 0)`, `process.nextTick()`, and `Promise.then()`?

### 1. `process.nextTick()`
Executes on the **current iteration** of the event loop, immediately after the current operation ends. It always runs before `setTimeout` and `setImmediate`.

### 2. `setTimeout(..., 0)`
Very similar to `setImmediate()`. Both run in the **next iteration** of the event loop. The exact execution order between the two can depend on various factors (like whether they are called within an I/O cycle).

---

## Task Queues & Execution Priority

Node.js manages different types of task queues with specific priorities:

1.  **`process.nextTick` Queue**: Highest priority. Executed after the current operation.
2.  **Promises Microtask Queue**: Executed after the `nextTick` queue.
3.  **Macrotask Queue** (`setTimeout`, `setImmediate`): Executed after microtasks are cleared.

> [!NOTE]
> Flow: `Current Operation` → `nextTick Queue` → `Promises Microtask Queue` → `Macrotask Queue`.

---

## Execution Order Example

Here is a demonstration of how these queues interact:

```javascript
const baz = () => console.log('baz');
const foo = () => console.log('foo');
const zoo = () => console.log('zoo');

const start = () => {
  console.log('start');
  
  setImmediate(baz);
  
  new Promise((resolve) => {
    resolve('bar');
  }).then(resolve => {
    console.log(resolve);
    process.nextTick(zoo);
  });
  
  process.nextTick(foo);
};

start();
```

### Expected Output (CommonJS)
In a **CommonJS** environment, the output is:
```text
start
foo
bar
zoo
baz
```

**Step-by-Step Breakdown:**
1. `start()` is logged.
2. `foo()` is added to the `nextTick` queue.
3. `bar` is added to the Promise microtask queue.
4. `baz()` is added to the macrotask queue.
5. The call stack ends:
    - **nextTick queue** runs: calls `foo()`.
    - **Microtask queue** runs: calls `bar` logger, which adds `zoo()` to the `nextTick` queue.
    - **nextTick queue** runs again: calls `zoo()`.
    - **Macrotask queue** runs: calls `baz()`.

---

## Behavior in ES Modules (ESM)

In **ES Modules** (`.mjs` files), the execution order changes:

**Output:** `start bar foo zoo baz`

> [!IMPORTANT]
> This happens because ES Modules are loaded as an asynchronous operation. The entire script is essentially already wrapped in the promises microtask queue. When a promise resolves immediately, its callback is appended to the microtask queue, which Node.js attempts to clear before moving to any other queue.
