# JavaScript Asynchronous Programming and Callbacks

## Asynchronicity in Programming Languages

Computers are asynchronous by design. Asynchronous means that things can happen independently of the main program flow.

### How Computers Handle Multiple Tasks
In current consumer computers, every program runs for a specific time slot and then stops its execution to let another program continue. This cycle runs so fast that it's impossible to notice. We think our computers run many programs simultaneously, but this is an illusion (except on multiprocessor machines).

Programs internally use **interrupts**, a signal emitted to the processor to gain the system's attention.

> [!NOTE]
> It is normal for programs to be asynchronous and halt their execution until they need attention, allowing the computer to execute other things in the meantime. When a program is waiting for a response from the network, it does not halt the processor; it yields control until the request finishes.

### Synchronous vs Asynchronous Languages
Normally, programming languages are synchronous, but some provide ways to manage asynchronicity natively or through libraries. 
- **Synchronous by default:** C, Java, C#, PHP, Go, Ruby, Swift, and Python.
- **Handling Async:** Some of these languages handle async operations by using threads or spawning new processes.

---

## JavaScript's Nature

JavaScript is **synchronous by default** and **single-threaded**. This means that code cannot create new threads and run in parallel. Lines of code are executed in series, one after another:

```javascript
const a = 1;
const b = 2;
const c = a * b;
console.log(c);
doSomething();
```

### The Browser Connection
JavaScript was born inside the browser. Its main original job was to respond to user actions like `onClick`, `onMouseOver`, `onChange`, and `onSubmit`. 

**How could it do this with a synchronous model?**
The answer lies in the **environment**. The browser provides a set of APIs that handle this kind of functionality. More recently, **Node.js** introduced a non-blocking I/O environment to extend this concept to file access, network calls, and more.

## Callbacks

Since you cannot predict when a user will click a button, you define an **event handler**. This handler accepts a function (the callback) which is executed when the event triggers.

```javascript
document.getElementById('button').addEventListener('click', () => {
  // Item clicked
});
```

### What is a Callback?
A callback is a simple function passed as a value to another function. It is only executed when a specific event happens. This is possible because JavaScript has **first-class functions**, which can be:
1. Assigned to variables.
2. Passed as arguments to other functions (higher-order functions).

### Common Examples

#### 1. Page Load
It's common to wrap client code in a `load` event listener to ensure the page is ready:
```javascript
window.addEventListener('load', () => {
  // Window loaded - initialization code here
});
```

#### 2. Timers
```javascript
setTimeout(() => {
  // Runs after 2 seconds
}, 2000);
```

#### 3. XHR (XMLHttpRequest)
XHR requests accept callbacks by assigning a function to a property that triggers on state changes:
```javascript
const xhr = new XMLHttpRequest();
xhr.onreadystatechange = () => {
  if (xhr.readyState === 4) {
    if (xhr.status === 200) {
      console.log(xhr.responseText);
    } else {
      console.error('error');
    }
  }
};
xhr.open('GET', 'https://yoursite.com');
xhr.send();
```

## Handling Errors in Callbacks

Node.js adopted a very common strategy for error handling: **Error-first callbacks**.

1. The **first parameter** of the callback is the error object.
2. If there is no error, the object is `null`.
3. If an error exists, it contains the description and metadata about what went wrong.

### Example: Reading a File
```javascript
const fs = require('node:fs');

fs.readFile('/file.json', (err, data) => {
  if (err) {
    // Handle error
    console.error(err);
    return;
  }
  // No errors, process data
  console.log(data);
});
```

## The Problem with Callbacks

Callbacks are great for simple use cases, but they have a significant drawback: **Nesting**. Every additional callback adds another level of indentation, causing the code to become complicated quickly (often called "Callback Hell").

```javascript
window.addEventListener('load', () => {
  document.getElementById('button').addEventListener('click', () => {
    setTimeout(() => {
      items.forEach(item => {
        // Your code here
      });
    }, 2000);
  });
});
```

While the example above is only 4 levels deep, real-world applications can have much deeper nesting, making maintenance difficult.

---

## Alternatives to Callbacks

Starting with ES6, JavaScript introduced features to handle asynchronous code more elegantly without deep nesting:

*   **Promises** (Introduced in ES6)
*   **Async/Await** (Introduced in ES2017)
