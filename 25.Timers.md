# Discover JavaScript Timers

## `setTimeout()`

When writing JavaScript code, you might want to delay the execution of a function. This is the job of `setTimeout`. You specify a callback function to execute later and a value expressing how much later you want it to run, in milliseconds.

```javascript
setTimeout(() => {
  // runs after 2 seconds
}, 2000);

setTimeout(() => {
  // runs after 50 milliseconds
}, 50);
```

### Passing Parameters

The syntax above defines a new function. You can call whatever other function you want in there, or you can pass an existing function name and a set of parameters:

```javascript
const myFunction = (firstParam, secondParam) => {
  // do something
};

// runs after 2 seconds
setTimeout(myFunction, 2000, firstParam, secondParam);
```

### Canceling a Timeout

`setTimeout` returns a value that can be used to cancel the scheduled function execution.

> [!NOTE]
> In **Node.js**, `setTimeout` returns a `Timeout` object instance. In **browsers**, it returns a numeric timer ID.

```javascript
const timeout = setTimeout(() => {
  // should run after 2 seconds
}, 2000);

// I changed my mind
clearTimeout(timeout);
```

---

## Zero Delay

If you specify the timeout delay to `0`, the callback function will be executed as soon as possible, but **after** the current function execution:

```javascript
setTimeout(() => {
  console.log('after');
}, 0);

console.log('before');
```

**Output:**
```shell
before
after
```

> [!TIP]
> This is especially useful to avoid blocking the CPU on intensive tasks. By queuing functions in the scheduler, you let other functions be executed while performing heavy calculations.

## `setInterval()`

`setInterval` is similar to `setTimeout`, with one key difference: instead of running the callback function once, it will run it forever at the specific time interval you specify (in milliseconds).

```javascript
setInterval(() => {
  // runs every 2 seconds
}, 2000);
```

### Stopping an Interval

The function above runs every 2 seconds unless you tell it to stop using `clearInterval`, passing it the interval ID that `setInterval` returned:

```javascript
const interval = setInterval(() => {
  // runs every 2 seconds
}, 2000);

clearInterval(interval);
```

### Auto-Stopping

It's common to call `clearInterval` inside the `setInterval` callback function to let it auto-determine if it should run again or stop.

```javascript
const interval = setInterval(() => {
  if (App.somethingIWait === 'arrived') {
    clearInterval(interval);
  }
  // otherwise do things
}, 100);
```

## Recursive `setTimeout`

`setInterval` starts a function every `n` milliseconds without considering when a function finished its execution.

### Scenario 1: Fixed Execution Time
If a function always takes the same amount of time, everything works fine:

```text
0         1000        2000        3000
|          |           |           |
|          |           |           |
|       [██████████]   |           |
|       [exe script]   |           |
|       [██████████]   |           |
|          |  [██████████]         |
|          |  [exe script]         |
|          |  [██████████]         |
|          |           |  [██████████]
|          |           |  [exe script]
|          |           |  [██████████]
|__________|___________|___________|____
```

### Scenario 2: Variable Execution Time
If the function takes different execution times (e.g., depending on network conditions), the intervals might not be what you expect:

```text
0         1000        2000        3000
|          |           |           |
|          |           |           |
|       [██████████]   |           |
|       [exe script]   |           |
|       [██████████]   |           |
|          |  [████████████████████]
|          |  [  execute script    ]
|          |  [████████████████████]
|          |           |     [██████████]
|          |           |     [exe script]
|          |           |     [██████████]
|__________|___________|___________|____
```

### Scenario 3: Overlapping Executions
In some cases, a long execution might even overlap the next one:

```text
0         1000        2000        3000        4000
|          |           |           |           |
|     [████████████████████]       |           |
|     [    execute script  ]       |           |
|     [████████████████████]       |           |
|          [███████████████████████████]       |
|          [      execute script       ]       |
|          [███████████████████████████]       |
|          |           |           [██████████████]
|          |           |           [execute script]
|          |           |           [██████████████]
|__________|___________|___________|___________|__
```

### The Solution: Recursive `setTimeout`

To avoid overlapping, you can schedule a **recursive `setTimeout`** to be called when the callback function finishes:

```javascript
const myFunction = () => {
  // do something
  setTimeout(myFunction, 1000);
};

setTimeout(myFunction, 1000);
```

This achieves a consistent gap between executions:

```text
0         1000         2000        3000
|          |            |           |
|     [█████████████]   |    [█████████████]
|     [ exec script ]>──1s──<[ exec script ]
|     [█████████████]   |    [█████████████]
|          |            |           |
|__________|____________|___________|__________
```

## Node.js Specifics

`setTimeout` and `setInterval` are available in Node.js through the `timers` module.

> [!IMPORTANT]
> Node.js also provides `setImmediate()`, which is equivalent to using `setTimeout(() => {}, 0)`. It is mostly used to work specifically with the **Node.js Event Loop** to ensure a callback is executed after the current poll phase completes.

> [!NOTE]
> Some browsers (IE and Edge) historically implemented a `setImmediate()` method, but it is not a standard web API and is unavailable in most modern browsers. In Node.js, however, it is a first-class citizen.
