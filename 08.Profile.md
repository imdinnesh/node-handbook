# Profiling Node.js Applications

Profiling involves measuring your application's performance by analyzing CPU, memory, and other runtime metrics. This process is essential for identifying:
*   **Performance Bottlenecks**: Identifying slow functions.
*   **High CPU Usage**: Pinpointing resource-intensive tasks.
*   **Memory Leaks**: Detecting inefficient memory management.
*   **Latencies**: Understanding impact on scalability and responsiveness.

While many third-party tools exist, the Node.js **built-in profiler** is often the easiest and most accessible option.

---

## The Built-in Profiler (V8)
The built-in profiler uses V8's internal profiling engine, which samples the stack at regular intervals. It records these samples as "ticks," along with optimization events like JIT compilation.

Example raw tick output:
```bash
code-creation,LazyCompile,0,0x2d5000a337a0,396,"bp native array.js:1153:16",0x289f644df68,~
code-creation,LazyCompile,0,0x2d5000a33940,716,"hasOwnProperty native v8natives.js:198:30",0x289f64438d0,~
code-creation,LazyCompile,0,0x2d5000a33c20,284,"ToName native runtime.js:549:16",0x289f643bb28,~
code-creation,Stub,2,0x2d5000a33d40,182,"DoubleToIStub"
code-creation,Stub,2,0x2d5000a33e00,507,"NumberToStringStub"
```

Historically, interpreting these ticks required the V8 source code. Since Node.js 4.4.0, built-in tools allow us to consume this data easily.

## Case Study: Performance Bottleneck in Express

Let's use a simple Express application to demonstrate. We have two handlers: one for creating users and one for authentication.

### 1. The Setup (Synchronous Logic)

```javascript
// NEW USER HANDLER
app.get('/newUser', (req, res) => {
  let username = req.query.username || '';
  const password = req.query.password || '';
  username = username.replace(/[^a-zA-Z0-9]/g, '');
  
  if (!username || !password || users[username]) {
    return res.sendStatus(400);
  }
  
  const salt = crypto.randomBytes(128).toString('base64');
  const hash = crypto.pbkdf2Sync(password, salt, 10000, 512, 'sha512');
  users[username] = { salt, hash };
  res.sendStatus(200);
});

// AUTH HANDLER
app.get('/auth', (req, res) => {
  let username = req.query.username || '';
  const password = req.query.password || '';
  username = username.replace(/[^a-zA-Z0-9]/g, '');
  
  if (!username || !password || !users[username]) {
    return res.sendStatus(400);
  }
  
  const { salt, hash } = users[username];
  const encryptHash = crypto.pbkdf2Sync(password, salt, 10000, 512, 'sha512');
  
  if (crypto.timingSafeEqual(hash, encryptHash)) {
    res.sendStatus(200);
  } else {
    res.sendStatus(401);
  }
});
```

> [!WARNING]
> **Disclaimer:** These handlers are for illustration only. Never design your own cryptographic authentication logic; always use proven solutions like Passport or dedicated libraries.

---

### 2. Generating a Profile
To investigate high latency, run the application with the `--prof` flag:

```bash
NODE_ENV=production node --prof app.js
```

Benchmark the server using **ApacheBench (ab)**:

```bash
# Register a user
curl -X GET "http://localhost:8080/newUser?username=matt&password=password"

# Put load on the /auth endpoint
ab -k -c 20 -n 250 "http://localhost:8080/auth?username=matt&password=password"
```

#### Benchmarking Results:
```text
Concurrency Level:      20
Time taken for tests:   46.932 seconds
Requests per second:    5.33 [#/sec] (mean)
Time per request:       3754.556 [ms] (mean)
```
The results show we are only serving ~5 requests per second, with an average latency of nearly 4 seconds.

---

### 3. Processing Trace Logs
The run generated a file named `isolate-0xnnnnnnnnnnnn-v8.log`. To make it human-readable, use the built-in tick processor:

```bash
node --prof-process isolate-0xnnnnnnnnnnnn-v8.log > processed.txt
```

#### Analyzing `processed.txt`:
Look at the **[Summary]** section first:
```text
 [Summary]:
   ticks  total  nonlib   name
     79    0.2%    0.2%  JavaScript
  36703   97.2%   99.2%  C++
      7    0.0%    0.0%  GC
    767    2.0%          Shared libraries
```
Since 97% of the samples are in **C++**, the bottleneck is likely in the Node.js core or a native module rather than your JS logic.

Checking the **[C++]** section:
```text
 [C++]:
   ticks  total  nonlib   name
  19557   51.8%   52.9%  node::crypto::PBKDF2(...)
   4510   11.9%   12.2%  _sha1_block_data_order
   3165    8.4%    8.6%  _malloc_zone_malloc
```
The `PBKDF2` function (used for password hashing) accounts for over half of the CPU time.

---

### 4. Bottom-Up Profile (Tracing the Caller)
To see what JavaScript code triggered these C++ calls, check the **[Bottom up (heavy) profile]**:

```text
   ticks parent  name
  19557   51.8%  node::crypto::PBKDF2(...)
  19557  100.0%    v8::internal::Builtins::~Builtins()
  19557  100.0%      LazyCompile: ~pbkdf2 crypto.js:557:16
```
This confirms that the `pbkdf2` call in our crypto module is the primary culprit.

---

### 5. Implementation: Asynchronous Optimization
The issue is that `pbkdf2Sync` is **synchronous**, which blocks the event loop and prevents other requests from being handled. Switching to the **asynchronous** version allows Node.js to handle other tasks while waiting for the crypto work to complete on a background thread.

```javascript
app.get('/auth', (req, res) => {
  let username = req.query.username || '';
  const password = req.query.password || '';
  username = username.replace(/[^a-zA-Z0-9]/g, '');
  
  if (!username || !password || !users[username]) {
    return res.sendStatus(400);
  }
  
  // Using the asynchronous version
  crypto.pbkdf2(
    password,
    users[username].salt,
    10000,
    512,
    'sha512',
    (err, hash) => {
      if (users[username].hash.toString() === hash.toString()) {
        res.sendStatus(200);
      } else {
        res.sendStatus(401);
      }
    }
  );
});
```

---

### 6. Verification
Running the benchmark again with the asynchronous code:

```text
Requests per second:    19.46 [#/sec] (mean)
Time per request:       1027.689 [ms] (mean)
```

**Results:**
*   **Throughput**: Increased from 5 to ~20 requests per second (4x improvement).
*   **Latency**: Dropped from 4 seconds to ~1 second.

## Conclusion
The V8 tick processor is a powerful built-in tool that provides deep insights into your application's execution. By identifying heavy C++ functions and tracing them back to your JavaScript code, you can make data-driven decisions to optimize performance.
